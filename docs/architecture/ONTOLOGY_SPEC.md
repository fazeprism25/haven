# Purpose

The Ontology subsystem exists to improve retrieval quality by organizing
KnowledgeObjects into a continuously evolving Concept Graph.

The ontology is NOT the primary storage layer.

KnowledgeObjects remain the source of truth.

The ontology is an indexing layer used only for retrieval.

---

# Design Philosophy

Manager AI answers:

"What happened?"

Ontology answers:

"What is this about?"

Memory Engine answers:

"What information is relevant right now?"

Each subsystem has a single responsibility.

# Core Principles

1. KnowledgeObjects are immutable.

2. Concepts are stable semantic entities.

3. Relationships connect Concepts.

4. KnowledgeObjects support Concepts.

5. The ontology never replaces KnowledgeObjects.

6. Retrieval begins from Concepts, not raw memories.

7. Downstream Memory Engine interfaces remain unchanged.

---

# Core Objects

## KnowledgeObject

Represents immutable knowledge extracted from conversations.

Acts as evidence.

Owns truth.

---

## Concept

Represents a stable semantic entity.

Examples:

- Haven
- Claude
- DTU
- Memory Engine
- Benchmark
- Luna

A Concept does NOT contain knowledge.

It organizes knowledge.

---

## Relationship

Represents a typed edge between Concepts.

Allowed relationship types:

- IS_A
- PART_OF
- USES
- DEPENDS_ON
- CREATED_BY
- LOCATED_IN
- RELATED_TO
- SUPPORTS

Relationships are deterministic.

---

## Attachment

Represents evidence linking a KnowledgeObject to one or more Concepts.

A single KnowledgeObject may support multiple Concepts.

Attachments prevent Concepts from owning KnowledgeObjects directly.

---

## Ontology Proposal

Represents proposed ontology changes generated by the Ontology Manager.

Examples:

- Create Concept
- Create Relationship
- Attach KnowledgeObject

Proposals never modify the graph directly.

---

## Ontology Validator

Validates ontology proposals before they modify the graph.

Responsibilities:

- duplicate detection
- alias resolution
- relationship validation
- deterministic IDs
- conflict prevention

---

## Concept Graph

Stores:

- Concepts
- Relationships
- Attachments

Contains no reasoning logic.

# Graph Expansion

Retrieval uses activation spreading.

Each activated Concept propagates activation to neighboring Concepts.

Propagation decays with distance.

Expansion stops when activation falls below a configurable threshold.

Retrieved KnowledgeObjects are collected from activated Concepts.

# Version 1 Scope

Implement:

- Concept model
- Relationship model
- Attachment model
- Proposal model
- Validator
- Concept Graph
- Markdown serialization
- Concept-aware CandidateRetriever
- Basic activation spreading

Do NOT implement:

- automatic ontology restructuring
- graph embeddings
- community detection
- background workers
- autonomous evolution
- reinforcement learning
- graph compression

---

# Success Criteria

The ontology is successful if it:

1. Improves retrieval quality.

2. Reduces retrieval search space.

3. Requires minimal modification to existing Memory Engine code.

4. Preserves deterministic retrieval.

5. Produces measurable benchmark improvements over keyword retrieval.

Every future ontology feature must improve retrieval or remain out of scope.
