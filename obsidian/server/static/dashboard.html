Jot a Markdown note. Haven saves it to your active Memory Space and runs it through the same memory pipeline as your ChatGPT conversations.

📥 Import Obsidian vault

Scan a folder of Markdown notes and import them into your active Memory Space (below in Settings), through the same memory pipeline as your conversations. This adds notes to the space you're already using — it does not create a new one. Want them kept separate instead? Create a new Memory Space in Settings first, then import here. Scanning never calls the LLM, and your original notes are never modified.

A custom vault folder can't be entered on a remote deployment — it would point at your browser's machine, not the server's. Scan always uses the active Memory Space's root here.

Hosted Demo Instance

This deployment contains preloaded demo Memory Spaces demonstrating Haven's capabilities. Local filesystem folders are intentionally unavailable on hosted deployments — use the Memory Spaces list below to switch between them.

Browsers can't hand a folder-picker dialog an absolute filesystem path, so paste one directly. Haven creates the folder (and its vault//concepts//.haven/ subfolders) if it doesn't exist yet, and never touches anything else already there.

Local folder selection isn't available here. Haven is running on a remote server, so a path typed into this browser (e.g. C:\Users\you\MyVault or /home/you/MyVault) points at your machine, not the server's, and can't be used as a Memory Space root.

Local filesystem paths only work when Haven is accessed via localhost or 127.0.0.1. This deployment manages its Memory Space directory via environment variables instead.

Memory Spaces This deployment is env-configured — Memory Spaces are managed via environment variables, not the dashboard.

Each Memory Space is a fully independent Haven database — activating one changes everything on this dashboard at once. Creating a space here only registers it; to add notes to it, activate it and then use Import Obsidian vault or Quick Capture above.

Query Rewriting

Uses an LLM to rewrite vague or underspecified queries before retrieval, searching every rewritten phrasing alongside the original for better recall. Retrieval, ranking, and acceptance logic are unchanged either way; this only changes how many queries are searched. The Retrieval Inspector shows both the original and rewritten queries when enabled. Off by default; switching back to Off at any time reproduces Haven's original, LLM-free behavior exactly, with no restart required.

Backend: local  ·  LLM: Qwen Cloud (DashScope), when configured  ·  health
